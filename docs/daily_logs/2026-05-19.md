**What I built:**
- Added `AutoSaveService` class (`src/service/auto_save_service.hpp`, `src/service/auto_save_service.cpp`): a background thread that calls a `std::function<void()>` save callable every N seconds using `condition_variable::wait_for` with a predicate on an atomic stop flag. `stop()` sets the flag, notifies the condition variable, releases the lock, then joins. Destructor calls `stop()` as a safety net.
- Added `std::mutex` to `CsvClientRepository` and `CsvPolicyRepository` with `lock_guard` on every method that reads or writes the data vector. Mutex declared `mutable` so `const` methods (`save`, `find`) can lock it.
- Changed `findAll` return type from `const std::vector<...>&` to `std::vector<...>` by value in both interfaces (`IClientRepository`, `IPolicyRepository`) and both implementations.
- Added a `std::mutex` guard around the shared static `mt19937` engine in `generateUuid` (`src/domain/utils.cpp`) to eliminate the data race when called from both the main thread and the autosave thread.
- Wired `AutoSaveService` into `Application` as `std::optional<AutoSaveService>`, constructed only when `autosave_enabled` is `true` in config. `cmdExit` calls `stop()` before setting `running_` to `false`. `main.cpp` now passes the interval from config.
- Added a `silent` parameter to `cmdSave` so the autosave callback suppresses stdout output, preventing interleaving with user prompts.

**What I learned:**
- `condition_variable::wait_for` with a predicate on an atomic stop flag allows the thread to be interrupted immediately on shutdown rather than waiting for the full remaining interval to expire.
- Correct `stop()` ordering is: set flag, notify, release the control lock, then join. Releasing the lock before joining is required because the thread itself holds no lock during the save call, so join can proceed without deadlock.
- The control mutex is released before calling save so the repository I/O does not block the stop path. Two separate lock scopes: one for the wait loop, one for the join.
- Destructor calling `stop()` is the RAII pattern for thread lifetime: if the caller forgets to stop explicitly, the destructor handles cleanup and prevents the thread from outliving the object.
- `findAll` must return by value, not by `const` reference. A reference would outlive the lock scope and be invalidated by any concurrent write that triggers vector reallocation.
- `mutable` mutex in `const` methods: `save()` and `find()` are logically `const` from the caller's perspective but must lock a member that is not conceptually part of the object's visible state. `mutable` is the correct qualifier here, not a design compromise.
- A shared static `mt19937` engine is mutable global state. Two threads calling `generateUuid` concurrently is a data race on that engine. A `lock_guard` around the engine access resolves it without changing the function's interface.

**Architecture reasoning:**
`AutoSaveService` is stored as `std::optional<AutoSaveService>` in `Application`, not a pointer, because it is a conditional resource, present only when `autosave_enabled` is true. `std::optional` expresses "may or may not exist" as a value with no heap allocation and no null pointer risk.

The `silent` flag on `cmdSave` keeps the autosave path and the manual save path as one function. The alternative of a separate `silentSave` method would duplicate logic. The parameter is a single bool at the call site, and the only caller that passes `true` is the autosave callback.

`stop()` is called in `cmdExit` before `running_` is set to `false` and before `Application` goes out of scope. This guarantees the background thread finishes any in-progress save and exits before the repositories it depends on are destroyed.
