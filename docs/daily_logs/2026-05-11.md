**What I learned:**
- assert() takes a single expression. Using a comma (assert(cond, "msg")) is the comma operator: it discards the left side and evaluates the string literal, which is always truthy. The assert never fires. Use && to attach a message: assert(cond && "msg"). If cond is false, false && "msg" is false and the assert fires.
- static_assert takes two arguments separated by a comma: static_assert(condition, "message"). The comma there is a function argument separator, not the comma operator. static_assert with a comma is correct.
- Belt and suspenders: assert + throw on the same condition for destructive operations. Assert catches bugs in debug. Throw prevents data corruption in release where asserts are removed. Apply only to operations with irreversible side effects (delete, edit). Read-only operations use assert-only or return optional.
- try/catch has zero runtime cost on the success path. The cost only occurs when an exception is thrown. Catch at the controller boundary (the layer that can display errors and re-prompt), not inside service or domain layers.
- std::runtime_error vs std::invalid_argument: use runtime_error for I/O and infrastructure errors (file not found, file open failure). Use invalid_argument for validation errors (bad input data). Mixing them means callers catching one type silently miss the other.
- A switch on an enum with no default is UB in C++ if no case matches and the function has a non-void return type. The compiler may warn but compiles it. The fix is a default that throws or asserts so the UB surface is eliminated.
- Embedding a newline (\n) in an exception message is wrong because different catch sites may format output differently. The catch site controls spacing and line endings.
- When wrapping a stdlib call like std::stod that gives a generic "invalid argument" message, always catch and rethrow with the actual value that caused the failure. "invalid amount in CSV: abc" is actionable; "invalid argument" is not.
