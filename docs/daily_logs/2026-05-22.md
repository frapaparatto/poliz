**Architecture reasoning:**

- `expired_date` is stored on `Contract` rather than computed on read because it is derived at creation time from `signed_date + contract_years`. Storing it avoids recomputing it on every access and makes it explicit in the CSV, so a future change to the calculation rule would not silently alter existing records.
- `editContract` restricts editable fields to status and notes. This is the same rule applied to `PolicyService`: the fields that define the formal agreement (value, product, dates) are immutable once signed. The comment in the service method refers back to ADR-019 so the rationale is traceable.
- `promptContractStatus` returns `std::optional` so the service can detect "user gave no input" and default to DRAFT without the controller needing to know what the default is. The default lives in one place: the service.
- `confirmClient` was moved from `PolicyView` to `cli_helper` because it is not view logic, it is input logic. Views display data; this function reads a keypress. Moving it also allows `InteractionController::cmdAdd` to reuse it without creating a dependency on `PolicyView`.
- `resolveInteraction` follows the same pattern as `resolveClient` and `resolvePolicy`: search, if empty return nullopt, if one result skip the selection step, otherwise show a list. Consistent pipeline across all entity types.
