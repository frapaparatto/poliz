**Implementation notes:**
`displayOne` previously took only a `const domain::Policy&`, which has no client name, only a UUID. To surface the name, the signature was extended with a `std::string_view client_name` parameter.

The harder question was where to source the name. `resolvePolicy` in `cli_helper.cpp` already resolves the client internally (it needs it to fetch that client's policies), but was discarding it on return. Changing the return type to `std::optional<std::pair<domain::Policy, domain::Client>>` let callers destructure with `auto& [policy, client] = *result` and forward the full name to `displayOne` at zero extra cost, no second lookup needed.

The three commands that go through `resolvePolicy` (`cmdView`, `cmdEdit`, `cmdDelete`) all use the pair. `cmdSearch` builds its own `name_map` from `client_repo_` for the table view; a small `clientName` lambda reuses that map for the drill-down call to `displayOne`, keeping the pattern consistent.

- Updated `main()` to call `parseConfig()` at startup and pass the result to both commands.

**Implementation notes:**
`parseConfig` returns a default config, not an error,  when the file is absent. This keeps startup silent for users who never create a config file.
