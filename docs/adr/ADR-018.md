## ADR-018 - File Path Strategy: Directory-Based Multi-Entity Persistence

**Date:** April 2026
**Status:** DECIDED

### Context

The original implementation used a single filepath for client data.
With Policy and Interaction entities arriving, each entity needs its
own CSV file. The startup flow must handle multiple files without
requiring the user to specify three separate paths.

### Options Considered

**Option A: Single filepath, derive others**
User provides a directory path at startup. The application derives
all entity filenames from it: `<dir>/clients.csv`, `<dir>/policies.csv`,
`<dir>/interactions.csv`. One prompt, three files.

**Option B: One file per entity, independent paths**
Each repository gets its own filepath specified independently.
Three prompts or three config entries. Maximum flexibility but
unnecessary complexity for this project.

**Option C: Single file for all entities**
All entities serialized into one CSV with a type discriminator column.
Simpler file management but parsing becomes complex, entity-specific
schemas are awkward, and the file becomes difficult to read manually.

### Decision

**Option A.** Directory-based path derivation.

### Rationale

One prompt covers all entities. The directory structure is predictable
and human-readable. Each entity has its own file with its own schema,
keeping serialization simple. Adding a new entity means adding one
filename derivation, not changing the user flow.

### Startup Behavior

**New session:** user provides a directory path. If the directory does
not exist, create it. All three repositories start empty with filepaths
set to `<dir>/clients.csv`, `<dir>/policies.csv`, `<dir>/interactions.csv`.
No files created on disk until the first save.

**Load session:** user provides a directory path. Check directory exists.
Each repository tries to load its file independently. If a file exists,
load it. If a file does not exist, that repository starts empty. A CRM
with clients but no policies yet is a valid state, not an error. Throw
only on actual errors: malformed data, permission denied.

### File Validation

A utility function `isValidCsvFile(const std::filesystem::path& path)`
in `insura::utils` validates file existence and extension before loading.
The existing existence check in `load()` stays as a defensive safety net.

### Impact

main.cpp changes from prompting for a single filepath to prompting for
a directory. It creates all three repositories with derived paths.
`CsvClientRepository::load()` behavior changes: missing file returns
empty state instead of throwing. Throw is reserved for malformed data
and I/O errors.
