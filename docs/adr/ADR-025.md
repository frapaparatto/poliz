## ADR-025: Interaction Repository Uses Clone for Safe Object Retrieval

**Date:** May 2026
**Status:** DECIDED

### Context

The interaction repository stores objects polymorphically as
std::vector<std::unique_ptr<Interaction>>. Unlike Client and Policy
repositories which store objects by value and return std::optional
copies from findByUuid, the interaction repository cannot return
by value because copying through a base type causes object slicing:
an Appointment copied as an Interaction loses its time, duration,
and report fields.

Three options were considered for findByUuid.

### Options Considered

**Option A: Return raw pointer (const Interaction*)**

The caller receives a non-owning pointer into the repository's
internal vector. Simple, no copy cost. But the pointer can be
invalidated if another thread modifies the vector between the
find and the subsequent operation (insert, remove, or auto-save
iterating). Same dangling reference problem that was fixed in
findAll by returning by value. Unsafe with threading.

**Option B: Return a clone via virtual clone() method**

Each derived class implements a clone() method that returns
std::unique_ptr<Interaction> containing a copy of itself using
the copy constructor. The caller owns an independent copy. The
repository's vector is untouched. No dangling pointer risk.
Thread-safe by design because the copy is made while the mutex
is held, and after findByUuid returns the caller's copy has no
connection to the repository's internal data.

**Option C: Return std::optional<Interaction>**

Not possible. std::optional stores by value, causing object
slicing on polymorphic types.

### Decision

**Option B.** Virtual clone() method on Interaction base class,
implemented by Appointment and Contract.

### Implementation

Interaction declares a pure virtual clone:

    virtual std::unique_ptr<Interaction> clone() const = 0;

Each derived class overrides using the compiler-generated copy
constructor:

    std::unique_ptr<Interaction> Appointment::clone() const {
        return std::make_unique<Appointment>(*this);
    }

The copy constructor works correctly because Appointment and
Contract contain only strings, ints, doubles, and optionals.
No raw pointers or special resources require a custom copy
constructor.

findByUuid locks the mutex, finds the element, calls clone()
on the found object, and returns the new unique_ptr. The lock
releases when findByUuid returns. The caller works on a fully
independent copy with no lock held.

### Edit flow with clone

1. Service calls findByUuid, receives an owned clone
2. Service modifies the clone using setters
3. Service passes the clone to updateInteraction via std::move
4. updateInteraction finds the original in the vector, replaces
   it (move assignment deletes the old object via RAII), sets dirty

No shared state between find and update. No dangling pointer.
No slicing. Clean ownership at every step.

### Rationale

Clone adds one virtual method and one copy per find operation.
The copy cost is negligible for CLI-scale usage (one find per
user command). The safety guarantee (no dangling pointers, no
slicing, thread-safe without caller coordination) is worth the
cost. Raw pointers would require the caller to hold the
repository lock for the entire edit operation, blocking other
threads unnecessarily.

### Impact

Added clone() as pure virtual to Interaction, override in
Appointment and Contract. findByUuid returns
std::unique_ptr<Interaction> (nullptr if not found) instead of
std::optional. findByClientUuid returns
std::vector<std::unique_ptr<Interaction>> with cloned objects.
