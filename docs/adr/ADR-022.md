## ADR-022: Policy Display with Client Name Resolution

**Date:** May 2026
**Status:** DECIDED

### Context

Policies store client_uuid as an internal foreign key reference.
The UUID is not a user-facing identifier. When displaying policies,
the user needs to see the associated client's name to understand
which client the policy belongs to.

### Decisions

**resolvePolicy returns std::pair<Policy, Client>**

Previously resolvePolicy returned std::optional<Policy>. The caller
then had no access to the client that was already resolved during the
policy lookup. cmdView, cmdEdit, and cmdDelete all needed the client
name for display but had no way to get it without a second lookup.

Returning std::pair<Policy, Client> gives the caller both values from
a single resolution pass. The client is already found internally as
part of the resolve pipeline (resolveClient runs first, then policies
are filtered by client UUID). Returning it costs nothing extra.

**UUID to name map for list and search display**

PolicyView::displayAll receives an
std::unordered_map<std::string, std::string> mapping client UUIDs to
full names ("First Last"). The map is built once per display operation
from IClientRepository::findAll() in the controller before calling
the view. Constant-time lookup per policy row.

The map is built per-call, not cached. For CLI-scale data volumes
(hundreds to low thousands of records), building the map each time is
negligible. Caching would add state management complexity for no
measurable benefit.

**PolicyView::displayOne receives client_name as parameter**

The detail view takes std::string_view client_name as a second
parameter. The caller is responsible for providing the name, either
from the pair returned by resolvePolicy or from the name map. The
view stays stateless and has no dependency on repositories or services.

**No "Unknown" fallback in name resolution**

The data model guarantees every policy has a valid client_uuid. A
policy is created only after resolveClient succeeds. Cascade delete
removes all policies when a client is deleted. If a policy references
a non-existent client, the data is corrupt. The lookup asserts that
the client exists rather than silently displaying a fallback:

```cpp
auto it = name_map.find(p.getClientUuid());
assert(it != name_map.end() && "policy references non-existent client");
return it->second;
```

This follows the error handling strategy (ADR-020): a policy without
a valid client is a programmer error (data integrity bug), not a
runtime condition the user can cause.

### Impact

resolvePolicy return type changed from std::optional<Policy> to
std::optional<std::pair<Policy, Client>>. All callers (cmdView,
cmdEdit, cmdDelete) updated to destructure with structured bindings.
PolicyView::displayOne signature updated with client_name parameter.
cmdSearch drill-down uses a local lambda wrapping the name map lookup.
