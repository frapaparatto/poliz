## ADR-021: Testing Strategy

**Date:** May 2026
**Status:** DECIDED

### Context

The project had no automated tests. Manual end-to-end testing was the
only verification method. An error handling audit revealed that many
code paths (CSV parsing, service business rules, cascade delete) had
no automated coverage and bugs in those paths would be invisible until
a user triggered them.

### Decision

Four testing levels, ordered by priority and implementation sequence.
No mocking. All tests use real objects and real temporary files.

### Level 1: Unit tests (Catch2)

**Target**: domain layer and utils. Pure functions with no I/O or side
effects.

**Coverage**: strops (trim, lower, capitalize, contains), date validation
(isValidDate, isLeapYear, isDateAfter, calculateEndDate), general
utils (isValidEmail, isDigitsOnly, isValidPhone, generateUuid,
stringToOptional), status converters (all enum-to-string and
string-to-enum round-trips), calculateAmount (all 16 type/duration
combinations).

**Pattern**: raw input, call function, check output. No mocks, no files,
no setup.

### Level 2: Integration tests (Catch2 + temporary files)

**Target**: data layer (CSV repositories) and service layer (business
rules that touch the repository).

**Coverage**: CSV serialize/deserialize round-trip for Client and Policy
(all fields, optional fields empty and populated, edge cases).
Save-and-reload cycle (insert, save, construct new repo, load,
compare). Service business rules (duplicate email rejection, cascade
delete, addPolicy with calculated fields).

**Pattern**: create temp directory, construct real repositories, exercise
the production code path, verify results, clean up temp files. No
mocks. The repository interfaces exist for dependency injection and
future storage swaps, not for creating test doubles.

### Level 3: Load tests (Python + Faker)

**Target**: performance under realistic data volume. Seed script generates
CSV files with thousands of entries. Measure load(), save(), and
search operations with std::chrono timers. Keep separate from unit
tests.

### Level 4: Approval tests (deferred)

**Target**: CLI output verification via golden file comparison. Manual
end-to-end testing covers the CLI for current scope.

### No mocking rationale
**Source**: "I Mock Your Mocks" (Boot.dev, Lane Wagner)

Mocking the repository tests the mock, not the code. A mock returns
what it is programmed to return, proving only that the test was
written correctly. Real files exercise the actual serialize/deserialize,
file I/O, and parsing logic.

Mocking is justified for external services with rate limits, network
dependencies, or irreversible side effects (payment processing,
third-party APIs). None apply to local CSV file persistence.

### Catch2 conventions

Test names describe behavior, not function names. SECTION groups
related checks under one TEST_CASE. GENERATE with table for
parameterized data (enum-to-string mappings). CHECK for independent
assertions in the same section, REQUIRE for preconditions and single
assertions. REQUIRE_THROWS_AS for exception testing. Catch::Approx
for all double comparisons.
