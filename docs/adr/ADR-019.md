## ADR-019: Policy Business Rules and Fixed Pricing Model

**Date:** May 2026
**Status:** DECIDED

### Context

The Policy entity originally had all fields editable (type, start date,
end date, amount, status, notes). During implementation it became clear
that allowing unrestricted editing creates ambiguity: can a user change
a policy type after creation? Can they manually set an end date that
contradicts the duration? Can they enter any amount?

These open questions produce edge cases that are expensive to validate
and test. A real insurance company offers fixed products with predefined
terms. The CRM should reflect that.

### Decision

Constrain the policy model to simulate a company that offers fixed
insurance products. The user selects from predefined options, the
system calculates the rest.

### User inputs at creation time

The user provides four things when adding a policy:

1. Client (resolved via existing search pipeline, stored as client_uuid)
2. Policy type (selected from enum: AUTO, LIFE, HOME, HEALTH)
3. Duration (selected from fixed options: 6, 12, 24, 36 months)
4. Start date (entered in YYYY-MM-DD format, validated by isValidDate)

### System-calculated fields

The system derives the remaining fields:

**End date:** start date + duration in months. Calculated by
`calculateEndDate(const std::string& start_date, int duration)`
in `insura::utils::date`.

**Amount:** looked up from a fixed pricing table based on type and
duration. No user input. The table represents the company's product
catalog.

### Pricing table

Based on Italian insurance market data (approximate annual ranges
simplified into fixed values for the CRM scope):

| Type   |  6 months | 12 months | 24 months | 36 months |
|--------|-----------|-----------|-----------|-----------|
| AUTO   |   350.00  |   650.00  |  1200.00  |  1700.00  |
| LIFE   |   150.00  |   280.00  |   520.00  |   740.00  |
| HOME   |   120.00  |   220.00  |   400.00  |   570.00  |
| HEALTH |   200.00  |   380.00  |   700.00  |  1000.00  |

These values are stored as a lookup table inside
`policy_service.cpp::calculateAmount`. The function
`calculateAmount(PolicyType type, int duration)` returns the
corresponding amount. If the combination is invalid, it throws.

Rationale for the values: AUTO is the most expensive category, consistent
with Italian RC Auto premiums averaging 350 to 500 EUR annually for
standard drivers, rising with duration. LIFE is moderate, reflecting
term life policies for healthy adults at 200 to 400 EUR annually for
basic coverage. HOME is the cheapest, matching Italian building insurance
at 120 to 250 EUR annually. HEALTH sits between LIFE and AUTO, reflecting
supplementary health coverage in the 200 to 400 EUR annual range.

### Multi-Year Scaling and Business Logic Simplification

*   **Linear Pricing Model:** For the scope of this CRM implementation, the 24-month and 36-month pricing tiers are calculated using a simplified linear projection based on the annual baseline. 
*   **Exclusion of Discounts:** This model explicitly omits advanced business logic such as multi-year commitment discounts, loyalty incentives, or promotional price reductions. 
*   **Exclusion of Dynamic Risk Factors:** In the real Italian market, multi-year policies (especially for `AUTO`) are uncommon due to annual adjustments in the *Bonus-Malus* risk rating system. This project intentionally bypasses dynamic risk factors, individual driver histories, and localized regional premium variations to prioritize a deterministic, predictable system architecture that simplifies testing and edge-case validation.

### Immutable after creation

Once a policy is created, the following fields cannot be changed:

- Policy type
- Start date
- Duration (not stored separately, implied by start and end date)
- End date
- Amount
- Client UUID

These define the contract. Allowing changes would invalidate the
business relationship between type, duration, and amount.

### Editable after creation

Only two fields can be modified through editPolicy:

- Status (active, expired, cancelled, pending)
- Notes (free text)

### Impact on implementation

**PolicyService::addPolicy:** receives PolicyData with type, client_uuid,
start_date, and duration_months. Calls calculateAmount and calculateEndDate
to derive amount and end_date. Constructs the Policy with all fields.

**PolicyService::editPolicy:** only accepts status and notes changes.
Rejects modifications to any immutable field.

**PolicyController::cmdAdd:** prompts for client (via resolveClient),
type (enum menu), duration (numbered menu: 6/12/24/36), start date
(validated string). Displays the calculated amount and end date for
confirmation before creating.

**PolicyController::cmdEdit:** prompts only for status and notes.
Does not display or allow modification of type, dates, or amount.

**PolicyData DTO:** includes type, client_uuid, start_date,
duration_months as required fields for add. For edit, only
optional status and optional notes.

### Duration field

Duration is not stored as a separate field on the Policy entity.
It is an input parameter used to calculate end_date and amount.
Once calculated, the duration is implicit in the difference between
start_date and end_date.
