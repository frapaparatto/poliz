## ADR-017 - Application Refactoring: Orchestrator + Controller Pattern

**Date:** April 2026
**Status:** DECIDED

### Context

The Application class was originally designed as a monolithic class
handling Client CRUD directly. With Policy and Interaction entities
coming, adding their commands, services, helpers, and dispatch logic
to the same class would make it unmanageable. The menu system also
needed restructuring from a single-level command menu to a two-level
entity selection plus command menu.

### Options Considered

**Option A: Prefix commands in existing menu**
Keep the monolithic Application. Add commands like "add-policy",
"list-policies" alongside existing client commands. No structural
change. Ships fast but the menu grows linearly with each entity
and Application becomes a massive class with three sets of CRUD
handlers, three sets of helpers, and three dispatch tables.

**Option B: If/else branching with direct references**
Application holds references to all three services. A "current entity"
string determines which service methods to call via if/else branches
in each command handler. No new classes but every command handler
contains branching logic and Application still owns all the
implementation details.

**Option C: Orchestrator with controller interface**
Application becomes a thin orchestrator that owns controllers via
a map of unique_ptr<IEntityController>. Each controller (ClientController,
PolicyController, InteractionController) implements the same interface
and owns its own service reference, dispatch table, and command
implementations. Application selects the active controller based on
user input and delegates all CRUD operations to it.

### Decision

**Option C.** Orchestrator pattern with IEntityController interface
and concrete controllers per entity.

### Rationale

Option A does not scale. Three entities with six commands each plus
helpers would produce a 1000+ line Application class with no
separation of concerns.

Option B reduces the class count but scatters entity-specific logic
across branching statements. Adding a fourth entity requires touching
every command handler.

Option C separates concerns cleanly. Each controller is self-contained
with its own service, helpers, and dispatch table. Adding a new entity
means creating a new controller class and one line in Application's
constructor. Application itself never changes when entities are added.

The IEntityController interface is justified because all three entities
share the same six CRUD operations with identical signatures. The
interface is small (six methods plus destructor) and stable.

Using unique_ptr for controller ownership is required because controllers
are accessed through the base interface. Storing derived objects by value
as the base type would cause object slicing. unique_ptr preserves the
full derived object and expresses sole ownership by Application.

This pattern also provides early practice with polymorphic ownership
via unique_ptr before the Interaction hierarchy (Appointment, Contract)
where the same pattern is required for domain objects.

### Menu Structure

Two-level menu:

- Level 1 (init menu): clients, policies, interactions, save, exit
- Level 2 (entity menu): add, list, search, view, edit, delete, back

"back" returns to level 1. Save and exit are application-level
commands handled directly by Application, not by controllers.

### Impact

Application class shrinks from ~400 lines to ~80 lines. All
client-specific code moves to ClientController. `promptRequired` and
`promptOptional` move to `cli_helper.cpp` as free functions in
`insura::cli`, shared across all controllers. `promptStatus` and the
status options array remain in `client_controller.cpp`'s anonymous
namespace. Menu gains displayInitMenu() and display() is renamed to
displayEntityMenu().
