## ADR-007: CSV Schema Design

**Date:** March 2026
**Status:** DECIDED

### Context

The CRM uses plain CSV files for persistence, one file per entity
(clients.csv, policies.csv, interactions.csv). Every design choice
about delimiters, column layout, and optional field encoding
determines how the serialization and deserialization code is written
and tested. These choices are not reversible without migrating existing
data files.

### Options Considered

**Option A: Alternative delimiter**

Use a delimiter that is unlikely to appear in field values (tab, pipe,
semicolon). Eliminates the need for quoting. Simpler parser.

The problem: none of these alternatives are guaranteed to be absent
from free-text fields. Tab characters can appear in copy-pasted text.
Pipe and semicolon appear in addresses and notes. The problem is moved
rather than solved.

**Option B: Comma delimiter, commas prohibited in field values**

Use comma as the delimiter. Prohibit commas in free-text fields as a
design constraint. Parsing reduces to std::getline with comma as the
delimiter. No quoting, no escaping, no multiline fields.

The trade-off: field values cannot contain commas. A client note that
reads "called Monday, discussed renewal" would need to be reworded.
This is an explicit constraint imposed on user input, not a silent
data loss.

### Decision

**Option B.** Comma as delimiter, commas prohibited in free-text
field values.

### Rationale

std::getline(ss, field, ',') is the parsing primitive used across all
three repositories. It handles every column, required and optional,
with identical code. A quoting layer would require a different parsing
path for quoted fields, which adds a branching parser and more test
surface.

The comma prohibition is a known limitation, not a hidden one. It is
documented here and in the architecture notes. For a CRM at this
scope, the constraint is acceptable. The serialization code is correct
and the integration tests verify round-trip fidelity for all fields.

Note: commas are not stripped or rejected by the domain setters in the
current implementation. The prohibition is enforced by convention.
A user who enters a comma in a free-text field will corrupt that row
in the CSV. This is a known gap, deliberately deferred in favor of
simpler code.

### Column layout

**clients.csv** (14 columns):

    uuid, first_name, last_name, email, phone, job_title, company,
    address, city, postal_code, status, notes, created_at, updated_at

Address is stored as three separate columns (address, city,
postal_code) rather than a combined string. This avoids embedding a
structured value in a single field and keeps individual address
components addressable without string splitting.

**policies.csv** (10 columns):

    uuid, client_uuid, policy_type, start_date, end_date, amount,
    policy_status, notes, created_at, updated_at

Amount is serialized with std::fixed and std::setprecision(2) to
avoid floating-point representation drift across save/load cycles.

**interactions.csv** (variable columns by type):

Appointment (10 columns):

    uuid, client_uuid, type, date, time, duration, report, notes,
    created_at, updated_at

Contract (12 columns):

    uuid, client_uuid, type, date, value, product_name, signed_date,
    expired_date, status, notes, created_at, updated_at

The type column (column 3, zero-indexed column 2) determines which
deserialization branch runs. Interaction rows are not a uniform width;
the parser reads the type first and then dispatches to the correct
field sequence.

### Optional field encoding

Absent optional fields are serialized as empty strings using
value_or(""). On deserialization, stringToOptional(field) converts
an empty string back to std::nullopt and a non-empty string to
std::optional<std::string> with the value. This convention is uniform
across all repositories.

### Impact

All three repository serialize and deserialize methods depend on this
schema. The column order is part of the file format contract: changing
a column position or adding a column without a migration breaks
existing data files. Integration tests cover the full round-trip for
each entity to catch regressions in the schema.
