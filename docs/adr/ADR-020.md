## ADR-020: Error Handling Implementation and Testing Strategy

**Date:** May 2026
**Status:** DECIDED

### Context

The codebase had inconsistent error handling: service layer functions
threw exceptions for conditions that are programmer errors in the
current single-threaded CLI architecture (e.g., UUID not found after
resolution pipeline already verified existence). Repository functions
silently no-oped on invalid UUIDs. The Policy constructor used
std::runtime_error for invalid argument conditions while the rest
of the domain layer used std::invalid_argument. No runtime asserts
existed anywhere in the codebase.

An audit (docs/personal/error_handling_audit.md) categorized every
throw, assert, and optional return against the project's error
handling strategy.

### Decision

Three error handling mechanisms, each with a specific role:

- **assert() for programmer errors.** Preconditions that should never
be violated if the code is correct. Active in debug builds, removed
in release via NDEBUG. The practical test: if this check fails, does
it mean the programmer made a mistake? If yes, assert.
- **Exceptions for rare runtime errors.** I/O failures, constraint
violations (duplicate email), malformed external data. The program
can recover. Zero cost on the success path, expensive only when
thrown.
- **std::optional for expected absence.** A query returning no result
is a normal outcome, not a failure. findByUuid returning nullopt,
searchClients returning an empty vector.

### Belt and suspenders for destructive operations

For operations with irreversible side effects (delete, edit), both
assert and throw are used together:

```cpp
auto client = repo_.findByUuid(uuid);
assert(client.has_value() && "deleteClient: UUID must exist");
if (!client) throw std::invalid_argument("Error: No contact found");
```

The assert fires in debug builds to surface the bug immediately
during development. The throw survives in release as a safety net
for future callers that may bypass the resolution pipeline.

This matters because the service layer is caller-agnostic. Today
only the CLI controller calls deleteClient after resolveClient
guarantees the UUID exists. A future API controller would receive
UUIDs directly from HTTP requests with no resolution step. The
throw protects data integrity regardless of the caller.

The rule for when to apply belt-and-suspenders: whenever the
operation has side effects that cannot be undone. Delete removes
data, edit modifies data. A wrong UUID reaching these in release
with no guard silently corrupts or loses data. For read-only
operations (find, search), a wrong UUID returns nothing via
optional and no damage occurs. Assert-only is sufficient for
non-destructive internal consistency checks.

### Exception type consistency

All domain-layer validation errors use std::invalid_argument.
The Policy constructor previously used std::runtime_error for
date and amount validation, which was inconsistent with setters
and the rest of the domain layer. Changed to std::invalid_argument
so callers catching std::invalid_argument don't silently miss
constructor-thrown exceptions.

std::runtime_error is reserved for I/O and infrastructure errors:
file not found, file open failure, filesystem errors.

### Testing strategy

Four levels, ordered by priority:

- **Level 1: Unit tests (Catch2).** Pure logic, no I/O. Domain
layer functions (strops, date validation, status converters) and
service layer calculations (calculateAmount). Raw input, call
function, check output. No mocks.
- **Level 2: Integration tests (Catch2 + temporary files).** Data
layer CSV round-trips (serialize, save, load, deserialize, compare).
Service layer business rules using real CsvRepositories pointed at
temporary files. Tests verify the actual production code path, not
a mocked abstraction.
- **Level 3: Load tests (Python + Faker).** Performance validation
under volume. Separate from unit tests, run manually or in
dedicated CI step.
- **Level 4: Approval tests (deferred).** CLI output verification
via golden file comparison. Manual end-to-end testing covers CLI
for current scope.

### No mocking rationale

Repository interfaces (IClientRepository, IPolicyRepository) are
not mocked in tests. Tests use real CsvClientRepository and
CsvPolicyRepository with temporary files. Rationale:

Mocking the repository tests the mock, not the code. A mock
returns what it is programmed to return, proving only that the
test was written correctly. Real files exercise the actual
serialize/deserialize, file I/O, and parsing logic.

The repository is local, fast, has no credentials, no rate limits,
no external dependencies. Creating a temp file, writing, reading,
and deleting is simpler than building a mock class and produces
more valuable test results.

Mocking would be justified for external services with rate limits,
network dependencies, or irreversible side effects (e.g., payment
processing, third-party APIs). None of those apply to CSV file
persistence.
