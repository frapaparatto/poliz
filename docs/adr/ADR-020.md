## ADR-020: Error Handling Strategy

**Date:** May 2026
**Status:** DECIDED

### Context

The codebase had inconsistent error handling: service layer functions
threw exceptions for conditions that are programmer errors in the
current single-threaded CLI architecture. Repository functions silently
no-oped on invalid UUIDs. The Policy constructor used std::runtime_error
for invalid argument conditions while the rest of the domain layer used
std::invalid_argument. No runtime asserts existed anywhere.

### Decision

Three error handling mechanisms, each with a specific role:

- **assert() for programmer errors.** Preconditions that should never
be violated if the code is correct. Active in debug builds, removed
in release via NDEBUG. The practical test: if this check fails, does
it mean the programmer made a mistake? If yes, assert.
- **Exceptions for rare runtime errors.** I/O failures, constraint
violations (duplicate email), malformed external data. The program
can recover. Zero cost on the success path, expensive only when thrown.
- **std::optional for expected absence.** A query returning no result
is a normal outcome, not a failure. findByUuid returning nullopt,
searchClients returning an empty vector.

### Belt and suspenders for destructive operations

For operations with irreversible side effects (delete, edit), both
assert and throw are used together:

```cpp
auto client = repo_.findByUuid(uuid);
assert(client.has_value() && "deleteClient: UUID not found");
if (!client) throw std::invalid_argument("client not found");
```

Assert fires in debug to surface the bug during development. Throw
survives in release as a safety net for future callers that may
bypass the resolution pipeline (e.g., an API controller receiving
UUIDs directly from HTTP requests).

**The rule**: apply belt-and-suspenders whenever the operation has side
effects that cannot be undone. Delete removes data, edit modifies
data. Read-only operations (find, search) use assert-only or return
optional.

### Exception type consistency

std::invalid_argument for all domain-layer validation errors (bad
input data). std::runtime_error reserved for I/O and infrastructure
errors (file not found, file open failure). Mixing them means callers
catching one type silently miss the other.

### Exception message format

Messages are lowercase, no "Error:" prefix, no exclamation marks,
no embedded newlines. The catch site at the controller boundary adds
the "Error: " prefix for display. The what() string is a clean
reason: "email already in use", "file not found: /path/to/file".
